# WireCopy UX audit — verified world-class-UX gaps

Generated by a 20-path agent swarm (audit → adversarial verify → completeness critic). **96 verified flaws** (9 major, 57 minor, 30 polish). Each survived a skeptical re-check against the code.

## Major (fix first)

### [credentials-hitl] Credential removal deletes without confirmation
- **Path:** 
- **Problem:** Confirmed: HandleCredentialRemove calls repo.DeleteAsync(credential.Id) at CredentialCommandHandler.cs:290 with no confirmation prompt — once a domain resolves to a credential it is deleted immediately. The parallel, LESS destructive cache-clear path in SearchCommandHandler.cs:419-431 DOES gate on ConfirmationDialog.ConfirmAsync, so the codebase already has the pattern and the more destructive op skips it. Deletion is recoverable only by re-entering the password. Not a blocker: the user explicitly types `:cred rm domain.com` (self-directed, no auto-select), and the data is re-creatable, so it does not block shipping — but the inconsistency with the existing confirm pattern on a more sensitive operation makes it major.
- **Fix:** Before repo.DeleteAsync at CredentialCommandHandler.cs:290, call ConfirmationDialog.ConfirmAsync(ctx.InputHandler, "Remove Credential", $"Remove credential for {domain}?", palette, ct) (same component SearchCommandHandler uses) and only delete on confirmation.
- **Evidence:** `CredentialCommandHandler.cs:290 (no confirm); SearchCommandHandler.cs:419-431 (ConfirmationDialog used for less-destructive :cache clear)`

### [cross-cutting] Clipboard copy fails silently when OSC 52 unsupported
- **Path:** User generates podcast, sees 'Feed URL copied to clipboard' message at line 1144, but due to terminal not supporting OSC 52, URL is not actually copied. User pastes expecting the URL and gets stale clipboard contents or nothing.
- **Problem:** CopyToClipboardOsc52 writes ANSI OSC 52 sequences to stdout without checking success, and if the terminal doesn't support OSC 52 or clipboard access fails, the user gets no feedback that the copy operation didn't work—they believe the URL/path was copied when it wasn't.
- **Fix:** Add error handling to CopyToClipboardOsc52: either check for OSC 52 support first, or provide a fallback/user prompt when the operation fails. Always communicate success/failure via SetStatusMessage.
- **Evidence:** `/workspace/src/WireCopy.Infrastructure/Browser/CommandHandlers/PodcastProgressScreens.cs:1618-1622 — function writes OSC 52 sequence without any error detection or user feedback`

### [cross-cutting] Rapid button presses during collection operations cause concurrent modification
- **Path:** User rapidly presses 'd' multiple times to delete collection items. Both delete operations are queued and execute, removing more items than intended. Undo buffer becomes inconsistent.
- **Problem:** There is no debouncing or concurrency guard on collection modification operations (save, delete, reorder), so rapidly pressing 'd' to delete items, or 'Shift+J/K' to reorder, can cause multiple operations to queue and execute concurrently, leading to lost updates or inconsistent state.
- **Fix:** Add a flag to CommandContext (e.g., 'IsProcessingCommand') that is set during any collection operation and cleared only after the operation completes. Skip command execution if the flag is already set.
- **Evidence:** `/workspace/src/WireCopy.Infrastructure/Browser/CommandHandlers/CollectionCommandHandler.cs — no concurrency protection on delete/reorder operations`

### [cross-cutting] Index bounds not checked after concurrent collection item deletion
- **Path:** User is on last item in a 5-item collection (index 4). Presses 'd' to delete. Another operation concurrently deletes 2 more items. Index becomes 4, but count is now 3. Next render causes array index out-of-bounds exception or shows wrong item.
- **Problem:** If a collection item is deleted while the selected index points to it, the index is adjusted but bounds checking is not performed atomically, so on the next render, CollectionItemSelectedIndex could exceed the collection size without catching it, causing an out-of-bounds read.
- **Fix:** After any collection deletion, clamp CollectionItemSelectedIndex to the valid range: 'Math.Min(selectedIdx, Math.Max(0, collection.Items.Count - 1))'. Do this in DeleteItem handler and in the refresh path.
- **Evidence:** `/workspace/src/WireCopy.Infrastructure/Browser/CommandHandlers/CollectionCommandHandler.cs — deletion adjusts index but doesn't re-clamp bounds after refresh`

### [cross-cutting] Concurrent podcast generation and collection modification causes race condition
- **Path:** User starts podcast generation from a 10-item collection. While generation is processing items, user presses 'd' to delete the currently-selected item from the collection. Generation thread's foreach loop now iterates over a mutated collection, skipping or duplicating articles.
- **Problem:** While podcast generation is reading from a collection to process articles, the user can modify the collection (delete items, reorder, rename), which mutates the Collection object while the orchestrator is iterating over Collection.Items. This causes indexing errors or skipped articles.
- **Fix:** Take a snapshot copy of collection items at the start of podcast generation: 'var itemsToProcess = collection.Items.ToList()'. Do not allow collection modifications while generation is in progress (set a flag or take an exclusive lock).
- **Evidence:** `/workspace/src/WireCopy.Infrastructure/Browser/CommandHandlers/PodcastCommandHandler.cs — collection reference is passed directly without snapshotting items`

### [onboarding-discovery] Ambiguous 'c' key causes confusion between Reading List and Setup
- **Path:**

### [open-and-history] Selection and scroll position reset on back navigation
- **Path:** 
- **Problem:** Confirmed. NavigationService.NavigateTo (line 131) pushes HistoryEntry(_currentPage, _currentViewMode) WITHOUT capturing _selectedLinkIndex or _scrollOffset, and GoBack (lines 181-182) hard-resets both to 0. HistoryEntry record (line 867) only holds Page and ViewMode. So returning via back lands at the top of the list with selection 0 — the user loses their place in the original link list. This is a genuine, repeated, user-visible friction in the core open-and-return loop. Not a blocker though: the page is fully usable and the position is re-findable, so 'blocker' is overstated.
- **Fix:** Extend the HistoryEntry record to also carry SelectedLinkIndex and ScrollOffset. Capture the current values at every push site (NavigateTo line 131, GoBack line 176, GoForward line 204), and in GoBack/GoForward restore entry.SelectedLinkIndex/entry.ScrollOffset instead of zeroing _selectedLinkIndex/_scrollOffset. Clamp on restore in case the page's link count changed.
- **Evidence:** `/workspace/src/WireCopy.Infrastructure/Browser/NavigationService.cs:131,181-182,867`

### [wizard-preview] Uncaught model-call exceptions tear down the wizard with no recovery
- **Path:** 
- **Problem:** Partly confirmed, claim overstated. RunAsync has no try-catch around its model calls (SetupWizard.cs:80,123,140,161,251,785). One entry — the apply-strategy path — IS handled: its caller catches Exception, logs, and shows 'Apply failed · {message}' (StrategyChooserHandler.cs:178-182), and RunSetupWizardAsync's finally (521-531) restores the original tree and clears the overlay. But the other entry — Ctrl+L AI setup (RunSetupEntryAsync → RunAiSetupAndApplyAsync:896 → RunSetupWizardAsync:502) — has NO local try-catch, so a model/network exception propagates to the top-level loop catch at BrowserOrchestrator.cs:625, which renders an error and whose finally (638-654) disposes the browser session — the whole app exits. So a transient network blip mid-setup ends the entire session with no retry. The claim's 'stack trace / no recovery' is overstated (RenderError shows ex.Message, terminal state and tree are restored cleanly via finally), and one path is fully handled — hence major, not blocker.
- **Fix:** Wrap the SetupWizard.RunAsync call inside RunSetupWizardAsync (the single chokepoint for both entries) in try-catch that rethrows OperationCanceledException but catches general Exception, logs it, sets a user-facing status ('AI setup failed — check your connection and try again'), and returns new Result { Cancelled = true }. The existing finally already restores the tree/overlay, so the session continues instead of exiting.
- **Evidence:** `/workspace/src/WireCopy.Infrastructure/Browser/CommandHandlers/StrategyChooserHandler.cs:896 (unwrapped entry path) vs :178 (wrapped apply path); /workspace/src/WireCopy.Infrastructure/Browser/BrowserOrchestrator.cs:625-655 (loop catch disposes session)`
